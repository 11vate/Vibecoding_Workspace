This research identifies **completely free, no-API-key solutions** across all game development domains:

### Critical Insight: 2025 Vibecoding Landscape

In 2025, "vibecoding" has matured from Andrej Karpathy's original concept into a spectrum:
- **Vibe Coding (Pure)**: AI generates code without review - ideal for prototyping
- **AI-Assisted Engineering**: AI as collaborator with code review - production-ready

**Y Combinator Report (March 2025)**: 25% of startups have codebases that are 95% AI-generated, but senior CTOs report "development hell" from unreviewed AI code in production.

**Recommendation**: Use vibe coding for rapid prototyping, AI-assisted engineering for production.

### 4.4 THREE.js for 3D Games

#### **Performance Best Practices (2025)**

- **LOD (Level of Detail)**: Simplified models for distant objects

##### **Rendering**
- **Key bottlenecks**: Draw calls, polygon count, shader complexity, texture size
- **Impact**: Reducing draw calls = biggest performance gain

##### **Lighting**
- **Dynamic lights**: Performance-intensive
- **Environment maps**: Realistic lighting without performance hit

##### **Emerging 2025 Features**
- **WebAssembly integration**: Performance-critical features (physics, simulations)

### 6.2 IndexedDB for Game Data

#### **Why IndexedDB?**
- **Non-blocking**: Doesn't slow down main thread
- **Service worker compatible**: Works from front-end and worker
- **Best for**: Structured data, searchable data, NoSQL-like operations

#### **Helper Libraries**

##### **idb** (Jake Archibald)
- **Type**: Promise wrapper for IndexedDB
- **Size**: Tiny
- **URL**: https://github.com/jakearchibald/idb

##### **localForage**
- **Features**: Good for older browsers lacking IndexedDB support
- **Fallback**: Graceful degradation

### 6.3 Storage Best Practices

#### **Storage Strategy**
1. **Cache API**: Network resources (HTML, CSS, JS, media)
2. **IndexedDB**: Structured data, user-specific data, game state

#### **Managing Large Datasets**
- **Chunking**: Break large data into smaller pieces
- **Benefits**: Reduced memory usage, enhanced retrieval speed

#### **Persistent Storage**
```javascript
navigator.storage.persist()
```
- **Default**: Data can be evicted under storage pressure
- **Persistent**: Only cleared by user

#### **Storage Quotas (2025)**
- **IndexedDB/Cache API**: Up to 60% of total disk space

#### **Offline Synchronization**
- Store data while offline
- Detect online status
- Sync when connectivity restored

### 7.4 State Management Patterns

#### **Zustand vs Redux (2025)**

##### **Zustand** ‚≠ê Recommended for New Projects
- **Philosophy**: Minimal API, no boilerplate, TypeScript-first
- **Performance**: Faster, lightweight
- **Use Case**: Smaller projects, avoiding boilerplate

##### **Redux Toolkit**
- **Features**: Predictable state container, middleware, dev tools
- **Performance**: Optimized but more overhead
- **Use Case**: Complex projects, large teams, rigorous patterns

##### **Decision Framework**
1. Start with **Context API** (simplest)
2. Graduate to **Zustand** (performance + structure)
3. Consider **Jotai** (complex interdependent state)
4. Adopt **Redux** (team size + complexity demand rigor)

#### **Game-Specific Patterns**

##### **Finite State Machines (FSM)**
- **Game State**: Loading, playing, paused, menu
- **Game Values**: Positions, lives, scores, level
- **Architecture**: FSM + publish-subscribe events

##### **State Pattern**
- **Benefit**: Object behavior changes with internal state
- **Implementation**: Assign state_ to different state objects

##### **State Stack Engine**
- **Use Case**: Layered game states (battle ‚Üí inventory menu)
- **Benefit**: Access different entities anytime

### Export Formats

1. **JSON** ‚≠ê Recommended for Web
   - Easier to load in browser

   - Standard format
   - Full feature support

   - Code export

   - Lightweight
